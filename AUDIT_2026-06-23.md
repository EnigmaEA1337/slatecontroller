# Audit nocturne — rapport multi-agents adversarial

_Run ID: wf_28a9dd09-d1f · 664 agents · 159 findings bruts → 12 après verify (92% refutés)_

## Executive summary

L'audit nocturne fait remonter une vulnérabilité critique : le défaut JWT_SECRET 'dev-secret-change-me' permet à la fois la forge de tokens admin et le déchiffrement Fernet de tous les secrets at-rest (clés SSH, RPC password, configs VPN, PSK WiFi), sans aucun garde au démarrage ni check hardening équivalent à celui de admin_password. Quatre findings high entourent ce cœur : absence totale de rate-limit sur /api/auth/login, vérification SSH host key désactivée (known_hosts=None) malgré le failover Tailscale et le pinning TLS déjà en place pour le RPC, export de la clé SSH privée derrière un simple GET sans PIN ni re-auth, et put_bytes/put_bytes_raw sans timeout qui peuvent figer indéfiniment plus de 30 call sites critiques. Les findings medium concernent la stabilité runtime (tâches background du lifespan jamais cancelled, leak sur reload) et le password admin Slate envoyé via pyglinet sur une session TLS non validée alors que le fingerprint est déjà en DB. Les findings low sont essentiellement des dettes de cohérence (race set_active, commentaire trompeur partial-push, no-op de filtrage .type dans le checker hardening) avec impact pratique limité. La thématique dominante reste 'la maison a les briques' — PinLockoutService, tls_fingerprint_sha256, anti-theft service existent déjà mais ne sont pas wired sur les surfaces sensibles correspondantes.

## Thèmes dominants

- Cryptographie / secrets (JWT_SECRET réutilisé pour signing et at-rest, length-leak compare_digest)
- Authentification (pas de bruteforce limiter, pas de jti revoke, export clé sans PIN)
- MITM / TLS / SSH (host key non vérifié, TLS verify désactivé sur RPC malgré pinning existant)
- Stabilité runtime (tâches background non cancelled, put_bytes sans timeout)
- Sync / cohérence (race set_active, partial-push mismatch commentaire/code)
- Hardening / firewall (faux filtre .type inflate le score)
- Adoption / device registry (TailscaleClient(None) si pas de default au boot)

## CRITICAL (1)

### 1. JWT_SECRET par défaut permet forge de tokens et déchiffrement de tous les secrets stockés

- **File**: `backend/app/config.py:35`
- **Category**: security
- **Theme**: Cryptographie / secrets

Settings.jwt_secret a comme valeur par défaut la chaîne litérale 'dev-secret-change-me' sans garde au démarrage. Ce même secret est utilisé pour signer les JWT HS256 (auth.py:61) ET pour dériver la clé Fernet via sha256 (vpn/crypto.py:30-32). Un attaquant qui n'a jamais changé l'env var peut forger des tokens admin {sub:admin,exp:future} ET déchiffrer trivialement tous les blobs stockés (DeviceSecretRow avec mots de passe RPC et clés SSH, VPNConfigRow.private_key_encrypted, mots de passe Fortinet, PSK WiFi, AppSecretRow). La jauge de hardening a un _controller_admin_password_check mais pas d'équivalent jwt_secret, donc un opérateur qui fixe ADMIN_PASSWORD garde une jauge verte. Aggravation : docker-compose.yml:37-40 expose le port 8000 sur le LAN au-delà du tailnet. Le couplage signing+at-rest sur un seul env var rend aussi la rotation destructrice (perte de tous les secrets chiffrés).

**Fix proposé** :
> Refuser le démarrage si jwt_secret est dans {'dev-secret-change-me','','change-me','change-me-to-a-long-random-string'} sauf flag DEV explicite. Ajouter un hardening check miroir de _controller_admin_password_check. Long terme : séparer la clé de signature JWT de la clé de chiffrement at-rest (deux env vars distinctes).

## HIGH (4)

### 2. Pas de protection bruteforce sur /api/auth/login

- **File**: `backend/app/api/routes/auth.py:23`
- **Category**: security
- **Theme**: Authentification

POST /api/auth/login accepte un nombre illimité de tentatives — pas de compteur IP/user, pas de backoff, pas de captcha. Avec le modèle single-admin et un JWT de 24h, un attaquant non authentifié peut faire du dictionary attack à la cadence native de FastAPI (centaines/sec). Le projet a déjà PinLockoutService (3 essais/60s) instancié sur app.state.pin_lockout mais wired seulement à /api/slate/screen-lock/verify. Le port 8000 publié sur le LAN augmente la surface au-delà du tailnet.

**Fix proposé** :
> Wrapper authenticate() avec PinLockoutService scope 'controller_login' keyé sur (request.client.host, username). Retourner 423/429 + Retry-After. Émettre un event structlog 'auth.login.throttled' distinct pour alerting SIEM.

### 3. Vérification SSH host key désactivée — pas de TOFU, pas de pinning

- **File**: `backend/app/slate/ssh.py:139`
- **Category**: security
- **Theme**: MITM / SSH

SlateSSH passe known_hosts=None à asyncssh.connect pour TOUS les devices. Or SlateUrlResolver fait du failover LAN↔Tailscale automatique et SSH._ensure_connected re-cible l'hôte à chaque reconnect — donc SSH passe transparemment par Tailscale. Le contrôleur fait pourtant déjà du TLS fingerprint pinning sur l'endpoint JSON-RPC (DeviceRow.tls_fingerprint_sha256) avec rejet 409 sur mismatch. Pire : sur Permission denied, _task_ensure_ssh_access (adoption.py:122-131) repush silencieusement la clé publique du contrôleur vers /etc/dropbear/authorized_keys de l'imposteur, après fallback password avec les creds root stockés.

**Fix proposé** :
> Stocker la pubkey SSH du serveur au premier connect réussi (TOFU), la persister dans DeviceRow à côté de tls_fingerprint_sha256, passer known_hosts=(known_keys,...) à asyncssh. Sur mismatch, fail loud avec un 409 'possible MITM or device re-flashed' identique au TLS. Rendre le pinning obligatoire pour les devices Tailscale-routed.

### 4. Clé SSH admin téléchargeable via simple GET auth-only — pas de PIN, pas de re-auth

- **File**: `backend/app/api/routes/settings.py:248`
- **Category**: security
- **Theme**: Authentification

GET /api/settings/ssh-keypair/private-key renvoie le PEM Ed25519 déchiffré derrière uniquement un JWT valide. Avec un token 24h et sans bruteforce limiter sur /api/auth/login, un token volé ou un mot de passe deviné donne la clé SSH root du Slate (qui peut être la seule credential restante si l'opérateur a désactivé l'auth password SSH côté Slate via settings.py:207-217). L'audit log à settings.py:266 ne capture que username — pas d'IP, pas de User-Agent — attribution post-incident faible.

**Fix proposé** :
> Gater l'export derrière PinLockoutService scope 'export_ssh_key' (réutiliser le pattern de slate.py:690). Ajouter request.client.host + user_agent dans l'audit log. Optionnel : URLs signées one-shot expirant en 60s bound à l'IP.

### 5. SlateSSH.put_bytes/put_bytes_raw sans timeout — un stdin remote bloqué fige indéfiniment

- **File**: `backend/app/slate/ssh.py:264`
- **Category**: error-handling
- **Theme**: Stabilité runtime

run() et run_binary() wrap conn.run() dans asyncio.wait_for(timeout=t). put_bytes() (l.264) et put_bytes_raw() (l.329) ne le font pas. Si le 'cat > file' remote stalle (disque plein, fs read-only, backpressure dropbear), la coroutine block sans borne. Le keepalive SSH (interval=30, count_max=3) ne protège que contre une TCP morte, pas contre un process remote qui draine lentement. Plus de 30 call sites concernés : deploy agent, push wallpaper, fortinet binary push, profile sync, screen_applier, fb_takeover. Plusieurs tournent en background FastAPI (finalize_after_reboot, surveillance) — un stall transient devient un task leak permanent.

**Fix proposé** :
> Wrapper les deux conn.run(..., input=payload, ...) dans asyncio.wait_for avec timeout caller-fourni ou cap (30-60s pour binary upload). Mirror exact du pattern run_binary.

## MEDIUM (2)

### 6. Tâches background du lifespan jamais annulées au shutdown

- **File**: `backend/app/main.py:441`
- **Category**: race
- **Theme**: Stabilité runtime

Le teardown du lifespan ne cancel que _tailscale_ha_task. _security_warmup_task, _ambient_scan_boot_task, _surveillance_boot_task et le refresh OUI fire-and-forget (référence non stockée) continuent à tourner pendant que device_registry.aclose_all() détruit les connexions SSH dont ces tâches dépendent. scheduler.shutdown(wait=False) n'attend pas non plus les jobs en vol. Résultat sur reload : SlateSSHError/asyncssh.ConnectionLost contre connexions droppées, écritures possibles sur engine half-disposed, warnings 'Task was destroyed but it is pending!' à chaque restart. L'OUI task leak génère aussi 'coroutine-was-never-awaited' si la loop ferme avant la fin.

**Fix proposé** :
> Tracker chaque tâche créée sur app.state, les cancel toutes dans le finally avant aclose_all(), puis await asyncio.gather(*tasks, return_exceptions=True). Idem pour le refresh OUI (stocker la référence).

### 7. Mot de passe Slate envoyé au JSON-RPC client avec TLS verification désactivée

- **File**: `backend/app/slate/client.py:296`
- **Category**: security
- **Theme**: MITM / TLS

_build_and_login instancie pyglinet avec verify_ssl_certificate=False et appelle gl.login() qui POST username+password sur cette session TLS non validée. Le contrôleur fingerprint-pin pourtant déjà le leaf TLS (DeviceRow.tls_fingerprint_sha256) mais ce pinning n'est consulté QUE dans /devices/{slug}/probe (devices.py:342). La voie RPC routine n'utilise jamais le fingerprint stocké — chaque reset de breaker ou re-login pyglinet retransmet le password admin sur un canal non validé. MITM LAN (ARP spoof contre 192.168.8.1, scénario mobile/OSINT du projet) capture le password à chaque reconnect.

**Fix proposé** :
> Après pyglinet build sa requests.Session dans _build_and_login, swap session.verify pour un callable qui valide contre tls_fingerprint_sha256, ou monkey-patch le post-connect transport check pour comparer SHA256 du leaf. Le fingerprint est déjà en DB — il suffit de le wirer.

### 10. authenticate() utilise bitwise & sur compare_digest ; mismatch de longueur leak toujours

- **File**: `backend/app/auth.py:47`
- **Category**: correctness
- **Theme**: Cryptographie / secrets

Le docstring prétend constant-time comparison pour éviter de leak username/password length via timing. secrets.compare_digest documente explicitement leak la longueur quand les arguments diffèrent en taille — l'early-out N'EST PAS timing-safe across lengths. Chaque appel compare_digest indépendant leak la longueur de son input. Le bitwise & garantit que les deux digests tournent (vs 'and' qui short-circuit), mais c'est un idiome non-évident qu'un futur mainteneur pourrait 'simplifier' en 'and' et briser l'intent constant-time. Single-admin MVP : risque pratique faible.

**Fix proposé** :
> Construire un token de forme fixe et comparer une fois : secrets.compare_digest(f'{username}\\0{password}'.encode(), f'{s.admin_username}\\0{s.admin_password}'.encode()). Supprimer la chaîne &.

### 11. set_active : race entre check d'existence et write AppState dans deux sessions séparées

- **File**: `backend/app/profiles/store.py:209`
- **Category**: correctness
- **Theme**: Sync / cohérence

ProfileStore.set_active(name) appelle d'abord self.get(name) (session A) puis ouvre session B pour upsert AppStateRow. Si le profil est supprimé entre A et B (concurrent DELETE), set_active écrit un pointeur active vers un nom inexistant. SQLite sérialise les writes donc la fenêtre est microseconde, et activate_profile (routes/profiles.py:898-904) appelle ensuite store.get(name) qui raise ProfileNotFoundError → 404. resolve_active_name in slate_agent/sync.py reconcile depuis le device, donc auto-heal. Impact réel : marqueur DB transitoirement stale.

**Fix proposé** :
> Inliner le check d'existence dans le même 'async with self._sf() as session:' que l'upsert, ou faire select(ProfileRow.id).where(ProfileRow.name == name) dans la même transaction et raise ProfileNotFoundError de l'intérieur.

### 13. JWT login sans iat et pas de revocation list

- **File**: `backend/app/auth.py:60`
- **Category**: security
- **Theme**: Authentification

create_access_token n'émet que sub + exp — pas d'iat, jti, audience, issuer. Le logout (auth.py:43-50) est explicitement stateless, log only, sans revoke. Avec un JWT 24h, un token volé ne peut être invalidé qu'en rotant jwt_secret (qui casse aussi Fernet — voir finding critical). Interaction mauvaise avec l'absence de bruteforce limiter sur /api/auth/login : un mot de passe bruteforcé donne 24h d'accès non révocable sauf redémarrage avec nouveau secret (qui force re-chiffrement de tous les secrets).

**Fix proposé** :
> Ajouter iat + jti et une deny-list jti persistante (AppStateRow keyée 'auth_revoked:<jti>' avec TTL jwt_expiration_hours). Logout ajoute le jti à la deny-list ; get_current_user vérifie après jwt.decode. Un DB read par requête authentifiée — pas cher et donne un vrai logout + 'sign out all sessions' sans rotation du master secret.
